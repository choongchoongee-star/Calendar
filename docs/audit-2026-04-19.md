# SPEC vs 구현 드리프트 감사 보고서

> 작성: 2026-04-19
> 범위: `SPEC.md` 기준 전체 구현 상태 재검증 (웹 + iOS 위젯 + Firestore + 메타데이터)

---

## 1. 감사 방법

SPEC.md의 검증 가능한 주장(§2 폴더 구조, §3 데이터 모델, §4 기능 명세, §5 API·UserDefaults 키, §7 App Store 체크리스트)을
실제 파일 내용과 1:1 대조하여 명확한 오류와 모호한 항목을 분리했다.

## 2. 검증 완료 — SPEC와 일치

| 영역 | 근거 |
| --- | --- |
| Firestore 스키마 (`calendars`, `schedules` 필드) | `firestore.rules` 주석 + `docs/script.js`의 실제 read/write 경로 |
| LocalStorage 키 (`isGuest`, `guest_calendars`, `guest_schedules`) | `docs/script.js` 전역 일치 |
| 위젯 family 사이즈 (`systemSmall`/`systemMedium`/`systemLarge`) | `CalendarWidget.swift:664` |
| 딥링크 스키마 (`vibe://`) | `ios/App/App/Info.plist` `CFBundleURLSchemes` |
| App Group (`group.com.dangmoo.calendar`) | `App.entitlements` |
| PrivacyInfo (UserDefaults CA92.1, FileTimestamp C617.1) | `PrivacyInfo.xcprivacy` |
| `ITSAppUsesNonExemptEncryption = false` | `Info.plist:48-49` |

| 공휴일 2025-2030 커버리지 | `docs/script.js` `getHolidaysForWeek()` + `CalendarWidget.swift` `holidayData` |

## 3. 명확한 오류 → 이미 수정 완료

### 3.1 `supabaseAuthToken` dead code 제거 (완료)
Phase 4 Firebase 마이그레이션 이후에도 위젯 브릿지에 `authToken` 파라미터가 JS→네이티브로
전달되고 있었고, 위젯은 `supabaseAuthToken` 키로 UserDefaults에 저장하고 있었음.
위젯 측 `getAuthToken()`은 정의만 돼 있고 **어디서도 호출되지 않음** (grep 전체 검색 0 건).

- **수정:** `docs/script.js:99` — `authToken` 필드 제거
- **수정:** `ios/App/App/WidgetBridge.swift` — `authToken` 읽기와 `supabaseAuthToken` 쓰기 제거
- **수정:** `ios/App/App/WidgetSource/CalendarWidget.swift` — `authTokenKey`, `getAuthToken()` 제거

### 3.2 SPEC §2 줄 수 현실화 (완료)
`script.js`: ~1,640 → ~1,690줄, `CalendarWidget.swift`: ~640 → ~680줄.

## 4. 확인 필요 (human review) — 커밋 전 사용자 확정

### 4.1 SPEC §2 폴더 구조 축약 정책
SPEC.md §2의 폴더 트리는 다음 iOS 파일을 생략하고 있다:
- `ios/App/App/AppDelegate.swift` (Capacitor 진입점)
- `ios/App/App/Info.plist`, `App.entitlements`, `PrivacyInfo.xcprivacy`
- `ios/App/App/WidgetBridge.m` (Capacitor ObjC bridge 등록)
- `ios/App/App/config.xml`, `public/` (Capacitor webDir mirror)
- `ios/App/exportOptions.plist`

**판단 필요:** 기획서 수준에서 의도적으로 생략한 것인지, 누락인지. 권장: "주요 파일만 표시"임을 명시.

### 4.2 `vibe://` URL scheme 이름
현재 Info.plist에 `vibe` 스키마가 등록돼 있다. 앱 정식 이름은 **채우다 (Chaeuda) / com.dangmoo.calendar**이므로
`vibe`는 레거시/개발 잔재로 보인다. 변경 시 위젯 딥링크 코드도 동시 수정 필요:
- `CalendarWidget.swift`의 `URL(string: "vibe://...")` 세 곳
- `docs/script.js`의 `appUrlOpen` 수신 핸들러 (만약 `vibe://` 체크가 있다면)

**판단 필요:** 그대로 유지 vs `chaeuda://`/`dangmoo://` 등으로 리네임. 리네임 시 사용자 기기 위젯은
재빌드·재설치 전까지 기존 딥링크를 계속 사용.

### 4.3 Google OAuth Client ID 하드코딩
`Info.plist:74-75`에 `GIDClientID`와 `CFBundleURLSchemes`에 Google reverse client ID가
그대로 커밋되어 있다. Google OAuth iOS Client ID는 Firebase 프로젝트·번들 ID에 묶인 공개값이지만,
정책에 따라 CI 주입으로 옮길 수도 있다.

**판단 필요:** Firebase 콘솔에서 이 Client ID가 iOS 번들 ID(`com.dangmoo.calendar`)에
제한된 경우 그대로 두고, 제한 없으면 CI 주입으로 이전.

### 4.4 Firebase team identifier 노출
`App.entitlements:14`에 Apple 팀 ID가 커밋돼 있다. Apple의 정상 관행이며 공개값이지만,
보안 감사 시 체크리스트로 언급됨.

**판단 필요:** 현행 유지(권장) vs CI에서 주입.

### 4.5 SPEC §4.1 "계정 탈퇴" 표기 vs 실제 동작
SPEC §4.1: "Firestore 데이터 삭제 후 `user.delete()`".
실제 구현에서 달력 소유자가 탈퇴할 때 해당 달력의 멤버들은 어떻게 되는지 기획서 불명확.
(현재 코드는 본인 소유 달력과 그 하위 schedule을 모두 삭제하지만, 멤버에게 공유된 달력의 경우
"본인이 소유자"라면 삭제, "본인이 멤버"라면 members map에서 자기 UID만 빠짐.)

**판단 필요:** SPEC §4.1에 "소유자 탈퇴 시 달력 전체 삭제, 멤버는 자동 탈퇴" 명시 여부.

### 4.6 2031+ 공휴일 정책
SPEC §9에 "2031+ 공휴일 - API 또는 수동 확장 필요"로 적혀 있으나 선택 시점과 방식이 정해져 있지 않다.
2030년 말까지 행동 트리거(예: CI 워크플로에서 만료 2년 전 경고)를 넣을지 여부.

**판단 필요:** 만료 경고 자동화 추가 vs 2030년 Q4에 수동 갱신 리마인더.

## 5. 권장 후속 작업 (감사 결과 기반)

1. §4.2의 `vibe://` 스키마 결정 → 리네임 시 한 PR로 (Info.plist + 위젯 + JS 핸들러)
2. §4.5 계정 탈퇴 정책 SPEC 한 줄 추가
3. §4.6 공휴일 만료 경고 GitHub Actions 워크플로 초안 (2028-12 기준 만료 2년 경고)
